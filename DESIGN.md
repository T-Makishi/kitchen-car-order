# キッチンカー事前注文アプリ 詳細設計

## 1. 目的と実装範囲

顧客がスマートフォンから商品を選択し、受取日時・場所を指定して予約注文できる公開画面と、店舗運営者が注文、商品、営業枠、店舗情報、通知を管理する認証済み管理画面を提供する。注文データは永続化し、注文確定後のメール失敗が注文保存へ影響しない構成とする。

実装先は既存資産と衝突しない `kitchen-car-order/` とする。既存のマラソン関連アプリや生成済み成果物には変更を加えない。

## 2. 採用技術と理由

- UI／ルーティング: Next.js App Router、React、TypeScript、Tailwind CSS
- 実行基盤: Sites 互換の Vinext／Cloudflare Worker ESM
- 構造化データ: Cloudflare D1（SQLite 互換）と Drizzle ORM
- 画像: Cloudflare R2。ローカル開発時は開発用ストレージを利用する
- 検証: Zod 相当のサーバー側スキーマ検証を境界ごとに実装する
- メール: SMTP 環境変数を受け取るメールアダプターを用意し、開発時はプレビュー記録、本番時は送信プロバイダーへ差し替え可能にする
- テスト: Node test runner による単体・結合テスト、ブラウザーによる主要導線確認
- 日時: 永続化は UTC ISO 8601、表示・営業時間判定は `Asia/Tokyo`

Prisma とローカルファイル SQLite ではなく D1／Drizzle を採用する。理由は、この環境で公開可能な Worker 実行基盤に適合し、D1 が SQLite 互換の永続ストレージであり、画像を R2 へ安全に分離できるためである。オンライン SMTP は Worker で直接ソケット接続できないため、メール機能をアダプター化し、開発プレビューと HTTP メールプロバイダーを実装対象とする。SMTP 設定項目は互換用に環境変数例と運用説明を残す。

## 3. アーキテクチャ

### 3.1 レイヤー

- `app/`: 顧客／管理画面、Route Handlers
- `components/`: 顧客用、管理用、共通 UI
- `lib/domain/`: 価格、営業時間、受取枠、状態遷移などの純粋関数
- `lib/server/`: D1、認証、CSRF、レート制限、メール、画像、監査ログ
- `db/schema.ts`: 全テーブル定義
- `drizzle/`: 生成済み SQL マイグレーション
- `public/`: PWA、代替画像、自作初期画像
- `tests/`: 単体・結合・主要画面テスト

### 3.2 主要処理フロー

1. 顧客は公開メニューを読み込み、ブラウザー内の一時カートへ追加する。
2. 注文確認時、サーバーが公開状態、売り切れ、在庫、購入上限、価格、営業日、締切、枠容量を再検証する。
3. クライアントが生成した冪等キーに一意制約を設け、同一送信は既存注文を返す。
4. D1 トランザクション相当のバッチで注文、明細、オプション、枠使用数、監査記録を保存する。
5. 保存完了後に顧客・店舗向けメール配送レコードを作り、メールアダプターで送信する。失敗時は `FAILED` とエラー概要だけを記録し、注文は維持する。
6. 完了画面は注文番号と別の確認トークンを使用する。注文番号単独では個人情報を返さない。

## 4. URL 設計

### 4.1 顧客画面

- `/`: 店舗情報、営業状況、本日の場所、お知らせ、おすすめ商品
- `/menu`: 検索、カテゴリー、並べ替え、商品一覧
- `/menu/[slug]`: 商品詳細、オプション、数量、備考
- `/cart`: カート編集、サーバー再検証、最終確認
- `/checkout`: 顧客情報、受取日時、場所、支払方法、同意
- `/orders/complete`: 完了情報
- `/orders/status`: 注文番号＋確認情報による状態照会
- `/offline`: オフライン案内
- `/privacy`, `/terms`: ポリシー

### 4.2 管理画面

- `/admin/login`: ログイン
- `/admin/setup`: 初回パスワード設定
- `/admin`: ダッシュボード
- `/admin/orders`: 一覧、検索、絞り込み、CSV
- `/admin/orders/[id]`: 詳細、修正、状態変更、印刷、メール再送
- `/admin/menu`: 商品・カテゴリー・オプション管理
- `/admin/schedule`: 場所、通常営業、臨時営業、受取枠
- `/admin/settings`: 店舗、通知、規約、パスワード、セッション設定

### 4.3 API

- 公開: `/api/public/store`, `/api/public/menu`, `/api/public/slots`, `/api/orders`, `/api/orders/status`
- 管理: `/api/admin/session/*`, `/api/admin/orders/*`, `/api/admin/menu/*`, `/api/admin/schedules/*`, `/api/admin/settings/*`, `/api/admin/uploads`, `/api/admin/email/test`
- 運用: `/api/health`, `/manifest.webmanifest`, `/sw.js`

管理 API はすべてサーバー側でセッション検証し、更新系は同一生成元確認と CSRF トークンを必須にする。

## 5. データ設計

金額はすべて日本円の整数。日時は UTC 文字列、日付と時刻指定は店舗タイムゾーンの文字列としても保持する。

- `admins`: ID、パスワードハッシュ、初期変更要否、失敗回数、制限期限、更新日時
- `admin_sessions`: ハッシュ化したトークン、CSRF 秘密、期限、最終利用日時
- `store_settings`: 店名、説明、連絡先、画像、SNS、通貨、税込表示、最低注文額、支払方法、メッセージ、規約、受付状態、準備時間、予約日数、セッション時間
- `business_locations`: 名称、住所、地図 URL、公開状態
- `business_schedules`: 曜日、場所、営業時間、受取間隔、枠上限、締切、当日可否
- `special_business_days`: 日付、営業／休業、場所、時間、告知
- `categories`: 名称、スラッグ、表示順、公開状態
- `menu_items`: 商品情報、税込価格、原材料、アレルゲン、辛さ、調理時間、在庫、上限、各種フラグ
- `menu_item_images`: オブジェクトキー、MIME、寸法、表示順
- `option_groups`: 必須、最小／最大選択数、表示順
- `option_choices`: 名称、追加料金、売り切れ、表示順
- `pickup_slots`: 場所、開始時刻、上限、予約数、停止状態
- `orders`: 推測困難な注文番号、確認トークンハッシュ、冪等キー、顧客情報、受取情報、支払、状態、合計、備考、申告、メール状態
- `order_items`: 商品名・単価スナップショット、数量、小計、備考
- `order_item_options`: グループ名・選択名・追加料金スナップショット
- `announcements`: 件名、本文、表示期間、重要度、公開状態
- `email_deliveries`: 種別、宛先マスク、件名、状態、試行回数、最終エラー、送信日時
- `audit_logs`: 管理者、操作、対象種別・ID、変更概要、IP ハッシュ、日時
- `rate_limits`: スコープ、識別子ハッシュ、期間、回数、期限

注文番号、冪等キー、セッショントークンハッシュに一意制約を設定する。検索対象、受取日時、状態、公開／表示順に索引を設定する。

## 6. 認証・セキュリティ設計

- パスワード: Web Crypto API で PBKDF2-SHA-256、ランダムソルト、実行基盤上限の反復回数。方式と反復回数をソルト文字列に記録し、平文を永続化・記録しない
- セッション: 256 bit 乱数、DB には SHA-256 ハッシュのみ、Cookie は HttpOnly／SameSite=Lax／本番 Secure／Path=/
- ログイン制限: IP と管理者の双方で失敗回数を計数し、一時ロックする
- CSRF: セッション紐付けトークン＋Origin/Host 検証
- XSS: React の標準エスケープを使用し、任意 HTML は描画しない。メール本文もエスケープする
- SQL: ORM／プリペアドステートメントのみ
- 注文: 公開 API をレート制限し、サーバー再計算、冪等性、一意制約を適用する
- 画像: JPEG／PNG／WebP の署名、MIME、5 MB 上限、寸法を検査し、UUID オブジェクトキーで保存する。配信時は `nosniff` と適切な Content-Type を付ける
- ログ: SMTP 資格情報、パスワード、完全な個人情報を記録しない
- エラー: 公開レスポンスは汎用メッセージ、内部例外はサーバーログへ限定する
- ヘッダー: CSP、frame-ancestors、Referrer-Policy、Permissions-Policy、nosniff
- 本番初期化: 未設定状態では管理 API を閉じ、初回セットアップ用ワンタイム秘密を環境変数で要求する

## 7. 注文ドメイン規則

- 合計 = Σ((商品単価 + 選択オプション単価合計) × 数量)
- 必須オプションは最小選択数以上、すべて最大選択数以下
- 非公開、売り切れ、在庫不足、商品別購入上限超過は拒否
- 最低注文額未満は拒否
- 受取日は営業日かつ予約可能日数以内、受取時刻は営業時間内・締切前
- 枠の予約済み件数＋注文数量または注文件数が上限を超える場合は拒否
- 状態遷移は `NEW → CONFIRMED → COOKING → READY → PICKED_UP`。各状態から許可された場合のみ `CANCELLED`。受取済みからの変更は不可
- キャンセル時は枠利用数と在庫引当を戻す

## 8. UI・アクセシビリティ設計

- モバイル幅を基準に、顧客画面下部へカート導線を固定する
- 44 px 以上の操作領域、16 px 以上の入力文字、明確なフォーカスリング
- 暖色系のブランドカラーと濃色テキストで WCAG 2.2 AA のコントラストを確保する
- 状態は色だけで表現せず、ラベルとアイコン相当の記号を併用する
- フォームエラーは項目直下と上部要約へ表示し、`aria-invalid`、`aria-describedby`、ライブリージョンを付与する
- 管理表は小画面ではカード表示へ切り替える
- 過剰なアニメーションを避け、`prefers-reduced-motion` を尊重する

## 9. PWA・障害対策

- マニフェスト、192／512 px アイコン、テーマカラー、standalone 表示
- Service Worker は静的シェルとオフラインページだけをキャッシュし、注文・管理 API はキャッシュしない
- 注文送信前にオンライン状態を確認するが、最終判断はサーバーが行う
- タイムアウト時に同じ冪等キーで再照会・再送でき、重複注文を作らない
- `/api/health` はアプリ、D1、時刻を確認し、秘密情報を返さない

## 10. 初期データ

- 店舗: `まちの小さなキッチンカー`
- 営業場所: Instagramで当日の出店場所を告知
- カテゴリー: ごはん、軽食、ドリンク
- 商品: 6 品以上。CSS／ローカル生成の権利問題がないプレースホルダー画像を同梱
- オプション: ご飯量、トッピング、辛さ、ドリンクセット
- 売り切れ、おすすめ、新商品をそれぞれ含める
- 通常営業、臨時日、お知らせ、確認用注文を投入する

## 11. テスト設計

- 単体: 金額、オプション、営業時間、締切、枠上限、入力、状態遷移
- 結合: 注文成功、売り切れ、枠外、冪等性、管理ログイン、パスワード変更、商品編集、状態変更、メール失敗継続
- E2E 相当: 顧客注文、管理状態変更、画像付き商品作成、パスワード変更、未認証遮断
- 品質ゲート: 型検査・ビルド成功、全テスト成功、主要画面の実ブラウザー確認、依存関係監査

## 12. 実装上の前提

- 店頭支払いのみで、オンライン決済は `PaymentMethod`／`PaymentProvider` 境界だけを用意する。
- 在庫は注文時に引き当て、キャンセル時に戻す。
- 受取枠上限は初期版では注文件数で計数する。
- 税込価格を直接登録し、税計算による丸めは行わない。
- 日本国内の電話番号を想定し、記号を除いた 10〜11 桁を許可する。
- メールは開発環境で配送内容を DB に記録して管理画面から確認できる。
- 公開時の画像保存は R2、ローカル開発時の互換保存方式は実行基盤のエミュレーターに従う。

## 13. 完了条件

- 顧客が商品を選択し、正しい受取枠で注文を確定・照会できる
- 管理者が安全にログインし、注文・商品・営業・店舗設定を操作できる
- 画像、メール記録、CSV、印刷、監査、PWA、ヘルスチェックが機能する
- 要求されたテストとビルドを実行して成功する
- README に初回設定からバックアップ・復元・公開・トラブル対応までを記載する
