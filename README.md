# まちの小さなキッチンカー 事前注文アプリ

キッチンカー向けのメニュー閲覧、受取予約、注文受付、管理を一つにまとめたモバイル対応PWAです。顧客画面と管理画面を分離し、注文、注文時点の価格、通知履歴、監査履歴を永続保存します。

## 主な機能

- 顧客: 営業案内、検索・カテゴリー・並べ替え、商品オプション、カート、予約注文、注文状況照会
- 管理: 初回設定、安全なログイン、注文自動取得、状態変更、取消、CSV、印刷、商品作成・複製・削除・公開・売切・画像、店舗設定、パスワード変更
- 運用: PWA、オフライン案内、冪等な注文、メールプレビュー／外部メールAPI、監査記録、ヘルスチェック
- セキュリティ: PBKDF2パスワードハッシュ、安全なCookie、CSRF、ログイン一時制限、サーバー再検証、画像署名検査、セキュリティヘッダー

詳細な判断根拠とデータ設計は [`DESIGN.md`](DESIGN.md) を参照してください。

## 必要環境

- Node.js 22.13以上
- pnpm 11以上
- Sites／Cloudflare互換のD1とR2開発バインディング

このプロジェクトはNext.js App Router互換のVinextで動作します。データベースはSQLite互換のCloudflare D1、画像はR2です。公開基盤で同じ構成を利用できるため、ローカルファイルSQLite＋PrismaではなくD1＋Drizzleを採用しています。

## 1. アプリを起動する

```bash
cp .env.example .env.local
pnpm install
pnpm dev
```

ターミナルに表示されたローカルURLをブラウザーで開きます。顧客画面は `/`、管理ログインは `/admin/login` です。

初回起動時は必要なテーブルとサンプルデータが自動作成されます。正式運用前は `drizzle/0000_kitchen_car.sql` を環境のD1へ適用してください。

## 2. 初回の管理者IDとパスワードを設定する

1. `.env.local` の `ADMIN_SETUP_TOKEN` を32文字以上の推測困難なランダム文字列へ変更します。
2. `/admin/login` を開き、「初めて設定する」を選びます。
3. 設定トークン、管理者ID、12文字以上で英大文字・英小文字・数字を含む管理パスワードを入力します。
4. 設定後は、ホスト環境から `ADMIN_SETUP_TOKEN` を削除または別の無効な値へ変更します。

パスワードをソースコード、README、チャット、共有メモへ記載しないでください。本番環境で初回設定トークンが未設定なら管理者を作成できません。

## 3. 管理画面へログインする

1. `/admin/login` を開きます。
2. 初回設定で作成した管理者IDと管理パスワードを入力します。
3. 5回連続で失敗すると15分間ログインを制限します。
4. 作業後は右上の「ログアウト」を選びます。

セッションCookieはHttpOnly、SameSite=Laxです。HTTPS環境ではSecure属性も付加され、有効期限は初期状態で8時間です。

## 4. 店舗情報を設定する

1. 管理画面の「店舗設定」を開きます。
2. 店舗名、紹介、電話、店舗メール、注文通知先、最低注文金額、受付状態を入力します。
3. アレルギー注意事項、完了メッセージ、プライバシーポリシー、利用規約を確認します。
4. 「店舗設定を保存」を選びます。

変更は管理者と時刻を含む監査ログの対象です。顧客画面の固定ブランド表現を変更する場合は `components/CustomerApp.tsx` も更新してください。

## 5. 商品と画像を登録する

1. 「メニュー管理」を開きます。
2. 商品名、カテゴリー、説明、税込価格、原材料、アレルゲンを入力し、「新しい商品を作成」を選びます。
3. 一覧で公開へ切り替えます。
4. 「画像を選択」からJPEG、PNG、WebPの5MB以下のファイルを登録します。
5. 必要に応じて価格、売り切れ、公開状態を変更します。

アップロード時は拡張子ではなくファイル署名も検査し、ランダムな保存名を生成します。注文済みの商品を削除した場合、履歴保全のため物理削除せず非公開になります。商品オプションの初期データは `lib/catalog.ts` で管理します。

## 6. 営業場所と受取枠を登録する

初期データの受取場所は「当日の出店場所」とし、Instagramで確認する運用です。受取時間は11:30〜14:00の15分間隔を仮設定しています。正式な営業時間、場所、特別営業日を変更する場合は、次を同時に更新してください。

- `lib/catalog.ts` の `locations` と `createPickupTimes`
- `app/api/orders/route.ts` の営業時間サーバー検証
- `business_locations`、`business_schedules`、`special_business_days` の運用データ

この3箇所の不一致は誤受付につながるため、変更後に結合テストを追加してから公開してください。

## 7. 注文受付を開始する

1. 「店舗設定」で注文受付状態を「注文受付中」にします。
2. 公開商品、価格、売り切れ、最低注文金額、通知先を確認します。
3. 顧客画面をスマートフォンで開き、実際にテスト注文します。
4. 管理画面へ注文が表示され、メール配送記録が作成されたことを確認します。

## 店頭オペレーション

### 店頭注文

1. 店頭で注文を受け付けます。
2. 商品を調理します。
3. Square端末でクレジットカード、PayPayなどの決済を行います。
4. 決済完了後に商品をお渡しします。

店頭注文はSquare側で管理し、Webアプリへの二重入力は行いません。

### 事前注文

1. 注文通知メールまたは管理画面の「事前注文」で新着注文を確認します。
2. 「注文を確認」「調理を開始」「受渡し準備完了」の順に状態を更新します。
3. お客様の来店時にSquare端末で決済します。
4. 「Square決済・受渡し完了」を選び、商品をお渡しします。

Webアプリ内ではオンライン決済を行いません。支払いはすべて受取時のSquare端末決済です。

## 8. 受信した注文を確認する

- 管理画面は30秒ごとに新着注文を取得します。
- 注文番号、顧客、電話、受取日時・場所、合計、状態、アレルギー申告、通知状態を確認できます。
- 日常運用では受取時刻の早い注文から準備します。
- CSV出力は「CSV出力」、注文票は「印刷」を使用します。

## 9. 注文状態を変更する

1. 注文の「注文を確認」「調理を開始」「受渡し準備完了」「Square決済・受渡し完了」を順番に選びます。
2. 操作前の確認画面で注文番号と変更先を確認します。
3. 取消時は「取消」を選びます。

許可されていない状態の飛び越し、受取済み・取消済みからの再変更はサーバーが拒否します。状態変更時は顧客向け通知が作成されます。

## 10. パスワードを変更する

1. 「店舗設定」の「管理パスワード変更」へ進みます。
2. 現在のパスワードと新しいパスワードを入力します。
3. 変更後は全セッションが無効になるため、再ログインします。

## 11. メール通知を設定する

### 開発環境

`.env.local` の `EMAIL_MODE=preview` を使用します。外部送信せず、HTML本文とテキスト本文を `email_deliveries` に保存します。注文自体はメール処理より先に保存されるため、メール失敗時も失われません。

### 本番環境

Workerは一般的なSMTPソケットへ直接接続できないため、SMTP対応メールサービスのHTTP送信APIを使用します。

```dotenv
EMAIL_MODE=api
EMAIL_API_URL=https://your-mail-service.example/send
EMAIL_API_TOKEN=replace-with-secret-token
SMTP_FROM=まちの小さなキッチンカー <orders@example.jp>
ORDER_NOTIFICATION_EMAIL=makishi0520@gmail.com
```

注文通知の宛先は管理画面の「店舗設定」で変更します。`ORDER_NOTIFICATION_EMAIL` はデータベースの通知先が空の場合だけ使う予備値です。HTTP APIには `{ from, to, subject, text, html }` のJSONをBearer認証付きで送信します。利用するサービスの受信形式が異なる場合は `lib/server.ts` の `deliverEmail` アダプターだけを変更します。SMTP認証情報はアプリログへ出力しません。`SMTP_HOST`、`SMTP_PORT`、`SMTP_USER`、`SMTP_PASS` はメールサービス側のSMTP設定と運用資料を揃えるための互換項目です。

## 12. データをバックアップする

公開環境の管理機能からD1をSQLとしてエクスポートし、R2画像を別に同期します。例:

```bash
wrangler d1 export <DATABASE_NAME> --remote --output backup-$(date +%Y%m%d).sql
wrangler r2 object get <BUCKET_NAME>/menu/<OBJECT_KEY> --file backups/<OBJECT_KEY>
```

復元前に新しいD1データベースへ `drizzle/0000_kitchen_car.sql` を適用し、その後バックアップSQLを読み込みます。復元作業は既存データを上書きする可能性があるため、必ず別の検証環境でリストアテストを行ってください。

## テスト

```bash
pnpm test
pnpm lint
pnpm build
```

テスト対象は価格・オプション・営業時間・締切・枠上限・電話番号・HTMLエスケープ・注文状態遷移と、画面／注文APIの重要構成です。APIの結合確認は開発サーバー上で注文、重複送信、売り切れ、管理ログイン、状態変更を実行します。

## 本番公開

1. D1の論理バインディング `DB` とR2の `UPLOADS` を作成します。
2. `drizzle/0000_kitchen_car.sql` をD1へ適用します。
3. `.env.example` の値をホスト側シークレットとして設定します。
4. HTTPS、独自ドメイン、メール送信ドメインのSPF・DKIM・DMARCを設定します。
5. `pnpm build` とテストが成功した成果物をSitesへ公開します。
6. `/api/health` が `status: ok` を返すことを監視します。

## セキュリティ上の注意

- `.env.local`、秘密鍵、管理パスワード、設定トークンをGitへ追加しないでください。
- D1とR2は最小権限でバインドしてください。
- 注文番号だけでは個人情報を閲覧できません。6桁確認コードも必要です。
- 本番公開前に依存ライブラリ監査を行い、高重大度の問題を解消してください。
- 個人情報を含むバックアップの保存期限、暗号化、アクセス権を運用規程へ定めてください。
- 重大なアレルギー申告は自動判断せず、必ず店舗から確認してください。

## トラブルシューティング

- 管理画面へ入れない: Cookieを許可し、ロック時間経過後に再試行します。初回なら `ADMIN_SETUP_TOKEN` を確認します。
- 商品が注文できない: 公開・売り切れ・在庫・購入上限・最低金額・受取時刻を確認します。
- 画像を登録できない: JPEG／PNG／WebP、5MB以下かを確認し、R2の `UPLOADS` バインディングを確認します。
- メールが届かない: `email_deliveries` の状態とメールAPIの応答を確認します。注文データは維持されています。
- データベースエラー: D1の `DB` バインディングとマイグレーション適用を確認します。
- オフライン: 注文確定はできません。接続復旧後、同じカートから再送してください。

## 採用した初期前提

- 受取時にSquare端末で決済します。オンライン決済は実装しません。
- 税込価格を日本円の整数で登録します。
- 予約は翌日以降、11:30〜14:00、15分間隔、14日先までを基本とします。
- 枠上限は注文件数で数えます。
- メールは開発時プレビュー、本番時HTTPメールAPIです。
- 画像はローカル開発エミュレーター／本番R2に保存します。
